## 1. リスク一覧

### R-01: macOSでrdevにアクセシビリティ権限が必要

| 項目 | 内容 |
|------|------|
| **影響度** | 高 — 権限がないと入力監視が完全に動作しない |
| **発生確率** | 確実 — macOSのセキュリティポリシーにより必須 |
| **対象** | macOSユーザー全般 |

**原因**: macOSはセキュリティ上、アプリケーションのキーボード/マウスイベントの監視にアクセシビリティ権限 (TCC) を要求する。`rdev::listen` はこの権限なしでは `CGEventTapCreate` に失敗する。

**対策**:
1. **起動時の権限チェック**: デーモン起動時に入力監視のテストを行い、失敗した場合は明確なエラーメッセージを表示
2. **エラーメッセージ**: 「システム環境設定 → セキュリティとプライバシー → プライバシー → アクセシビリティ にこのアプリを追加してください」
3. **フォールバック**: 権限がない場合は `generate` サブコマンドのみ使用可能 (監視なしモード)
4. **ドキュメント**: READMEにmacOSでの権限設定手順を明記

**実装**:
```rust
fn check_input_permission() -> Result<()> {
    #[cfg(target_os = "macos")]
    {
        // テスト用のリスナーを短時間起動して権限を確認
        let (tx, rx) = std::sync::mpsc::channel();
        let handle = std::thread::spawn(move || {
            rdev::listen(move |_| { let _ = tx.send(()); })
        });
        match rx.recv_timeout(Duration::from_secs(2)) {
            Ok(_) => Ok(()),
            Err(_) => Err(anyhow!(
                "入力監視の権限がありません。\n\
                macOS: システム設定 → プライバシーとセキュリティ → アクセシビリティ \n\
                にこのアプリケーションを追加してください。"
            )),
        }
    }
    #[cfg(not(target_os = "macos"))]
    Ok(())
}
```

### R-02: Waylandでrdevが動作しない可能性

| 項目 | 内容 |
|------|------|
| **影響度** | 高 — Waylandユーザーで入力監視不可 |
| **発生確率** | 中〜高 — Wayland環境ではセキュリティモデル上、グローバルな入力キャプチャが制限される |
| **対象** | Wayland (Sway, GNOME Wayland等) ユーザー |

**原因**: Waylandのセキュリティモデルではクライアントが他のクライアントの入力を監視することを禁止している。`rdev` はLinuxでは `/dev/input/` デバイスファイルを直接読むか、X11のXInputを使用するが、Waylandのみの環境ではXInputが使えない。

**対策**:
1. **`/dev/input/` フォールバック**: `input` グループへの所属を前提に、evdevデバイスを直接読む
2. **libinput 経由**: `sudo libinput debug-events` のサブプロセスからイベントをパース
3. **権限要件の明示**: Wayland環境では `input` グループへの追加が必要であることをドキュメント化
4. **グレースフルデグラデーション**: 入力監視が不可能な場合は定期的な壁紙変更モード (タイマーベース) にフォールバック

**フォールバック実装**:
```rust
enum MonitorBackend {
    Rdev,           // デフォルト (X11, macOS, Windows)
    Evdev,          // Linux /dev/input/ 直接読み
    TimerOnly,      // 入力監視なし、タイマーベースで壁紙変更
}

fn select_monitor_backend() -> MonitorBackend {
    // rdev を試行 → 失敗なら evdev → それも失敗なら TimerOnly
}
```

**TimerOnlyモードの動作**:
- 設定可能な間隔 (デフォルト: 30分) で壁紙を自動変更
- 集中度分析は行わない
- ログに「入力監視不可のためタイマーモードで動作中」と記録

### R-03: 壁紙切替時のちらつき

| 項目 | 内容 |
|------|------|
| **影響度** | 中 — 視覚的な不快感、集中力支援の目的に反する |
| **発生確率** | 中 — バックエンドとOS設定に依存 |
| **対象** | 全プラットフォーム (特にswaybg) |

**原因**:
- 壁紙ファイルの書き込み途中に壁紙セッターが読み込む
- swaybg はプロセスの kill/spawn に伴い一瞬黒画面になる
- 一部の壁紙セッターはファイル更新を即座に反映しない

**対策**:
1. **アトミック書き込み**: 一時ファイルに書き込み完了後、`rename()` で目的ファイルに置換 (同一ファイルシステム上なら原子的)
2. **一意ファイル名**: 各フレームを異なるファイル名で出力し、現在表示中のファイルを上書きしない
3. **swaybg 専用処理**: swaybg 使用時はステップ数を削減 (25→10)、かつ `swaybg` の再起動ではなくシンボリックリンクの切り替えを試行
4. **遅延削除**: 前フレームの削除は次フレーム設定の500ms後に実行

**実装**:
```rust
fn write_frame_atomic(image: &RgbaImage, target: &Path) -> Result<()> {
    let temp = target.with_extension("tmp.png");
    image.save(&temp)?;
    std::fs::rename(&temp, target)?;
    Ok(())
}
```

### R-04: 集中力判定の誤検知 (偽陽性)

| 項目 | 内容 |
|------|------|
| **影響度** | 中 — 集中しているのに壁紙が変わると逆に気が散る |
| **発生確率** | 中 — 読書、動画視聴、考え込み中など |
| **対象** | 全ユーザー |

**シナリオ**:
- ドキュメントを読んでいる (マウス/キーボード操作なし = アイドルだが実は集中)
- 動画を視聴している
- 考え込んでいる (長時間の無操作)

**対策**:
1. **ヒステリシス**: 2回連続 (10分間) の低スコアが必要
2. **閾値の調整可能性**: ユーザーが `unfocused_threshold` を調整できる
3. **中立スコア**: イベントがない場合、各指標は0.5 (中立) を返す。これにより短時間のアイドルだけでは非集中と判定されない
4. **重み調整**: 読書が多いユーザーは `idle_weight` を下げることで対応

**将来的な改善案** (Phase 5以降):
- 前面アプリケーション情報の取得 (e.g., ブラウザ、IDE) により読書/動画視聴を判別
- 手動の「集中モード」トグル (一時的に監視を無効化)

### R-05: 4K画像生成のメモリ使用量

| 項目 | 内容 |
|------|------|
| **影響度** | 低〜中 — メモリ不足でOOM killerに殺される可能性 |
| **発生確率** | 低 — 通常のPCであれば問題ない |
| **対象** | メモリが少ないマシン (< 4GB) |

**原因**: 4K RGBA画像 = 3840 × 2160 × 4 bytes = 約33MB。トランジション中は2枚 + ブレンド結果で約100MB。

**対策**:
1. **メモリ見積もり**: 起動時に必要なメモリを見積もり、利用可能メモリが不足する場合は警告
2. **解像度フォールバック**: メモリ不足時は自動的に解像度を下げる
3. **ストリーミング書き出し**: ブレンド結果を全ピクセルメモリに持たず、行単位でPNGに書き出す (imageクレートのencoderを使用)

### R-06: ノイズ生成が想定より遅い

| 項目 | 内容 |
|------|------|
| **影響度** | 中 — トランジション中の待ち時間が長くなる |
| **発生確率** | 低 — rayonによる並列化で通常は高速 |
| **対象** | コア数が少ないマシン、古いCPU |

**対策**:
1. **オクターブ数の動的調整**: 生成時間が目標を超えた場合、次回からオクターブ数を1減らす
2. **プリレンダリング**: デーモン起動時に次の壁紙を事前生成しておく
3. **品質設定**: `generation.quality = "low" | "medium" | "high"` で簡単に性能/品質トレードオフを選択

**プリレンダリング戦略**:
```rust
// デーモン起動時に次の壁紙を事前生成
async fn prerender_next_wallpaper(config: &GenerationConfig) -> RgbaImage {
    tokio::task::spawn_blocking(move || {
        generate_nebula(config, None) // ランダムシード
    }).await.unwrap()
}
```

### R-07: rdevクレートのバグや非互換

| 項目 | 内容 |
|------|------|
| **影響度** | 高 — 入力監視が完全に動作しない |
| **発生確率** | 低 — rdevは安定したクレートだが、OSアップデートで破壊される可能性 |
| **対象** | 全プラットフォーム |

**対策**:
1. **rdevのバージョン固定**: `Cargo.lock` をリポジトリに含める
2. **パニックキャッチ**: `rdev::listen` のコールバック内で `std::panic::catch_unwind` を使用
3. **監視スレッドのリスタート**: 監視スレッドがクラッシュした場合、3回まで再起動を試行
4. **代替クレートの検討**: `device_query` をフォールバックとして使用可能に設計

```rust
fn start_monitoring_with_retry(buffer: Arc<Mutex<EventBuffer>>, max_retries: u32) {
    let mut retries = 0;
    loop {
        let buf = buffer.clone();
        let result = std::panic::catch_unwind(|| {
            rdev::listen(move |event| {
                // ...
            })
        });

        match result {
            Ok(Ok(())) => break, // 正常終了
            _ => {
                retries += 1;
                if retries >= max_retries {
                    tracing::error!("入力監視の再起動に{max_retries}回失敗。タイマーモードに切り替え");
                    break;
                }
                tracing::warn!("入力監視がクラッシュ。再起動 ({retries}/{max_retries})");
                std::thread::sleep(Duration::from_secs(5));
            }
        }
    }
}
```
